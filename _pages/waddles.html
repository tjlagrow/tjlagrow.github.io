Waddles AI Evolution Roadmap

Stage 1 (Legacy Fallback)

Static VSM Search

Keyword term-similarity index. Runs real-time text matching and extracts boundary-aligned preview text. Used as a fallback if the client browser does not support WebAssembly or fails to initialize neural models.

Stage 2 (Completed)

Local LLM & RAG

Quantized WebAssembly transformer models (ONNX/Transformers.js) in the browser. Uses all-MiniLM-L6-v2 to compute query vectors, runs browser-side cosine similarity against pre-computed embeddings, and feeds top contexts to SmolLM-135M-Instruct.

Stage 3 (Future)

Agentic API Tooling

Activating tool-use and function calling. Waddles will autonomously access external APIs to fetch live publication citation metrics, schedule office hour slots, and trigger verified contact notifications.

Client-Side Vector Database

Dense Semantic Embeddings

Sentence embeddings translate text snippets into 384-dimensional vectors. Geometrically, vectors that represent conceptually similar text (e.g. "resting-state fMRI" and "spatiotemporal dynamic patterns") point in similar directions, allowing Waddles to matching questions on conceptual intent rather than literal keyword occurrences. Waddles utilizes the all-MiniLM-L6-v2 transformer model to generate these semantic vectors.

Cosine Similarity Vector Matching

To rank document relevance without an external server, Waddles implements a direct cosine similarity calculation between the user's query vector and the pre-computed document chunks.

For query vector $q \in \mathbb{R}^{384}$ and document chunk vector $d \in \mathbb{R}^{384}$, the cosine similarity is defined as:

Similarity(q, d) = ∑i=1384 (qi × di)

Because the Python embeddings generator and the client-side transformers library extract vectors that are pre-normalized to unit length ($\|q\| = 1$, $\|d\| = 1$), the denominator of the cosine similarity equation ($\|q\|\|d\|$) is equal to 1. Thus, the similarity matching is calculated in pure JavaScript as a simple dot product, which evaluates in less than a millisecond.

1. Browser-Side Vector Extraction

When a user types a query, Waddles loads Hugging Face's Xenova/all-MiniLM-L6-v2 using Transformers.js to compute a 384-dimensional vector: 

const output = await embedder(query, {
  pooling: 'mean',
  normalize: true
});
const queryVector = Array.from(output.data);

This vector is matched against the cached embeddings database using the dot-product similarity, and chunks with similarity scores exceeding 0.38 are selected as the context window.

2. Local Language Synthesis (RAG)

Waddles then feeds the matched context chunks into a prompt compiled under the ChatML syntax, and runs a quantized local WebAssembly text-generator (onnx-community/SmolLM-135M-Instruct):

  • Prompt Injection: The context blocks constrain the generator model to prevent hallucinations.
  • Greedy Decoding: Running inference with temperature: 0.0 and do_sample: false ensures deterministic, factual responses.
  • Citations: Waddles dynamically appends source URL links mapping back to the page layout or research document that provided the facts.

1. Single-Threaded WASM Fallback

By default, ONNX Runtime Web requests multi-threaded WASM compilation, which requires SharedArrayBuffer support. For security (Spectre mitigations), browsers block this unless headers like Cross-Origin-Opener-Policy (COOP) and Cross-Origin-Embedder-Policy (COEP) are sent by the host. Since static hosts like GitHub Pages do not support custom headers, Waddles forces a single-threaded runtime configuration (wasm.numThreads = 1) to bypass this block, allowing compilation on any standard host.

2. Dynamic CDN Model Retrieval

To keep the website's Git repository light and performant, binary weights are not stored in the repository. Instead, Waddles dynamically retrieves pre-quantized ONNX configurations from the Hugging Face Hub CDN. Cross-Origin Resource Sharing (CORS) is enabled at the CDN level, allowing static pages to fetch files directly without triggering browser cross-origin blocks.

3. Persistent Browser-Side Caching

To preserve bandwidth and ensure zero-latency initial load times, Waddles leverages the browser's Cache Storage API (IndexedDB fallback). Because GitHub Pages serves pages over secure contexts (HTTPS), browser-side persistent caches are activated. Once downloaded, subsequent openings of the chatbot bypass the network, loading models locally and instantaneously.

4. Robust Search Mode Fallbacks

In the event that CDN download speeds are throttled, WebAssembly execution is blocked by administrative client-side policies, or older browsers are used, Waddles implements an automated fallback routing. The client-side logic catches any model load exceptions and falls back to our high-resolution exact-match static VSM keyword index, keeping the chatbot responsive.
