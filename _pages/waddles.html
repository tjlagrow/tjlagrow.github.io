Waddles AI Evolution Roadmap

Stage 1 (Current)

Static VSM Search

Zero-dependency term-similarity index. Parses the site into a minified JSON vector space, runs real-time TF weighting, and extracts boundary-aligned preview text. 100% client-side, zero latency, free hosting compute.

Stage 2 (Planned)

Local LLM & RAG

Deploying quantized WebAssembly transformer models (ONNX/Transformers.js) in the browser. Uses local dense vector embeddings to calculate true cosine semantic similarities and synthesizes page answers.

Stage 3 (Future)

Agentic API Tooling

Activating tool-use and function calling. Waddles will autonomously access external APIs to fetch live publication citation metrics, schedule office hour slots, and trigger verified contact notifications.

The Static Document Space Model

Traditional AI chatbots rely on large embedding models or API-based vector databases (like Pinecone or Chroma) which require server setups and cause latency.

Waddles bypasses this constraint by representing your website as a static vector index. During compilation, Jekyll creates a single array at /assets/json/search.json. The index strips out HTML markup, layouts, and style parameters, and structures the content as follows: 

{
  "title": "Page Name",
  "url": "/page-url/",
  "content": "Raw stripped text content...",
  "type": "page|post|newsletter"
}

The Similarity Scoring Formula

To rank document relevance without an external server, Waddles implements a custom term-frequency (TF) search algorithm in pure JavaScript.

For a normalized search query $Q$ tokenized into filtered keywords $\{t_1, t_2, \dots, t_n\}$ and a site document $D$:

Score(D, Q) = [ ∑ (TFtitle(ti, D) × Wtitle) + (TFcontent(ti, D) × Wcontent) ] × Coverage(Q, D)
  • $TF_{title}(t, D)$: Binary check (1 if query token $t$ is inside the page title, 0 otherwise).
  • $W_{title}$: Title boost multiplier (fixed at 20) to prioritize pages named after the topic.
  • $TF_{content}(t, D)$: The word-boundary frequency count of keyword $t$ in the content text (weighted at 2.5).
  • $Coverage(Q, D)$: Matched keywords divided by total query keywords. This penalizes documents matching only a single search term in longer questions (e.g. "Matthieu Bloch ECE course").

1. Tokenization & Punctuation Filtering

When a user types a query, Waddles normalizes it using string regex operations to remove punctuation and extract alphabetic words: 

const tokens = query.toLowerCase()
    .replace(/[^\w\s-]/g, '')
    .split(/\s+/)
    .filter(t => t.length > 2);

A built-in stopword dictionary filters out high-frequency connectives (e.g. the, with, about, what) that carry zero search intent. This prevents false positives and concentrates weighting on terms like cajal, 7641, or publications.

2. Boundary-Aligned Excerpt Windowing

When a matching document is selected, Waddles does not output the page's entire content. Instead, it runs a boundary-alignment previewer that locates where the matched term appears in the document:

  • Locates index of the first matching query keyword.
  • Carves out a text window (40 characters before, 120 characters after the term).
  • Finds nearest space boundaries (" ") at the start and end of the window to ensure words aren't truncated mid-letter.
